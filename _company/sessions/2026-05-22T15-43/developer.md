# 📄 기술 보고서 초안: LLM 시스템의 파일/상태 관리 메커니즘 고도화 방안

**작성 목적:** LLM 에이전트가 생성하는 파일 및 내부 상태 정보의 지속성, 신뢰성, 그리고 확장성을 확보하여, 실제 운영 환경(Production Grade)에서 사용할 수 있는 파일 관리 아키텍처를 제안함.

**작성자:** 코다리 (Senior Full-Stack Engineer)
**날짜:** 2024-05-23

## 1. 현행 시스템 분석 (Current State Analysis)

### 1.1. 현재 구현 방식 (Mechanism)
*   **사용 도구:** `<create_file>`, `<edit_file>`, `<read_file>` 등 시스템 인터페이스 태그.
*   **작동 원리:** 시스템 레벨에서 샌드박스 환경 내 가상 파일 시스템을 제공받아 텍스트 기반의 입출력을 시뮬레이션.
*   **장점:** 개발 및 디버깅 단계에서 즉각적인 코드 반영 및 테스트가 용이함.
*   **기술적 제약 (The Limitations):**
    1.  **격리성 (Isolation):** 현재 파일 시스템은 LLM 세션에 종속적이며, 외부 시스템(CI/CD, 사용자 DB)과의 직접적인 데이터 연동이 어려움.
    2.  **원자성 (Atomicity) 부족:** 대규모의 파일 변경이나 복합적인 상태 트랜잭션(Transaction)을 단일 작업 단위로 보장하기 어려움.
    3.  **데이터 포맷 강제성:** 모든 데이터가 '파일 내용'이라는 텍스트 형태로만 다뤄지므로, 구조화된 메타데이터(Schema) 관리가 어려움.

## 2. 아키텍처 개선 방안 (Proposed Solutions)

현재의 파일 시스템 접근 방식에 의존하는 대신, 데이터와 상태를 **'서비스 계층(Service Layer)'**을 통해 관리하는 방식으로 전환해야 합니다.

### 💡 방안 1: 전용 API 서비스 계층 도입 (가장 권장)
가장 근본적인 해결책은 파일 시스템 접근을 LLM의 핵심 로직에서 분리(Decouple)하는 것입니다.

*   **구현 목표:** 모든 파일 입출력 및 상태 저장을 전용 `StorageService` API를 통해 처리하도록 강제합니다.
*   **핵심 API 설계:**
    *   `storage.save(data: SerializedData, path: string, metadata: Object)`: 데이터를 저장하고, 파일 경로와 함께 메타데이터(버전, 작성자, 상태)를 함께 기록합니다.
    *   `storage.load(path: string)`: 경로를 통해 데이터를 로드하며, 데이터와 함께 모든 메타데이터를 반환합니다.
    *   `storage.get_history(path: string)`: 해당 파일의 모든 버전 히스토리를 조회합니다.
*   **기술적 이점:** 파일 내용을 단순히 텍스트로 취급하는 것이 아니라, **버전이 관리되는 구조화된 객체**로 취급하게 되어, Git과 유사한 트래킹(Tracking)이 가능해집니다.

### 💡 방안 2: 외부 NoSQL 데이터베이스 활용 (대안)
파일 내용을 시스템 파일 대신 외부의 강력한 상태 저장소에 보관합니다.

*   **구현 목표:** 모든 프로젝트의 상태와 생성된 파일은 전용 NoSQL DB (예: MongoDB, Redis)에 저장합니다.
*   **작동 방식:** LLM은 파일 내용을 생성할 때마다, 이를 JSON 또는 BSON 포맷으로 직렬화(Serialize)하여 DB에 저장하고, 시스템 인터페이스에는 DB Key(UUID)만을 노출합니다.
*   **적합 상황:** 대규모 병렬 작업이나, 세션 간의 상태 유지(Cross-Session State Management)가 매우 중요한 경우에 최적입니다.

### 💡 방안 3: 구조화된 데이터 출력 포맷 강제 (최소 수정)
외부 서비스 연동이 당장 불가능할 경우, 출력 포맷 자체를 강제합니다.

*   **구현 목표:** '파일'을 생성하는 대신, 모든 결과물을 **Canonical JSON Schema**로 출력하도록 강제합니다.
*   **예시:**
    ```json
    {
      "file_name": "config.json",
      "schema_version": "1.0",
      "data": {
        "api_key": "process.env.API_KEY",
        "endpoint": "https://api.example.com"
      },
      "description": "시스템 설정을 담고 있는 JSON 파일입니다."
    }
    ```
*   **제한 사항:** 이 방법은 파일의 '존재' 자체를 대체할 수 없으며, 단순히 '데이터의 형태'만 강제할 수 있습니다.
